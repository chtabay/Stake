# [Nom du projet]

> Infrastructure de dépôt, d'engagement et de résolution de takes.

---

# Profession de foi

La plupart des opinions publiques n'ont :
- aucun coût ;
- aucune traçabilité ;
- aucune obligation de cohérence ;
- aucune condition explicite de falsification.

Le projet propose une infrastructure permettant :
- de déposer une take ;
- d'en définir les conditions de résolution ;
- d'associer des preuves et protocoles de vérification ;
- d'engager publiquement une somme limitée ;
- d'organiser les conséquences prévues en cas de réussite ou d'échec.

Le projet ne cherche pas :
- à déterminer la vérité ;
- à promouvoir certaines opinions ;
- à maximiser l'engagement émotionnel ;
- à créer un système spéculatif.

Le projet cherche à permettre :
- l'engagement explicite ;
- la traçabilité ;
- l'archivage ;
- la résolution vérifiable d'hypothèses ;
- la mémoire des convictions dans le temps.

Chaque take devient :
- une capsule temporelle ;
- une hypothèse exécutable ;
- un engagement documenté ;
- une trace durable.

---

# Positionnement

La plateforme :
- héberge des takes ;
- stocke leurs conditions ;
- gère leur résolution ;
- gère les dépôts associés ;
- exécute les redistributions prévues ;
- archive les engagements et leurs preuves.

La plateforme ne :
- fait pas la promotion des takes ;
- ne cherche pas à créer un réseau social d'attention ;
- ne pousse pas certaines opinions ;
- ne fonctionne pas comme un média éditorial ;
- ne constitue pas un marché prédictif classique ;
- ne constitue pas un système de paris.

Son rôle est infrastructurel.

---

# Vision culturelle

Le projet repose sur l'idée suivante :

> une opinion structurée, falsifiable et assumée publiquement possède plus de valeur qu'une réaction instantanée sans conséquence.

Le système cherche à :
- ralentir la parole ;
- encourager les hypothèses explicites ;
- pousser à la formalisation ;
- archiver les visions du futur ;
- déplacer les débats vers les preuves ;
- créer une mémoire des convictions.

Le système valorise :
- la cohérence ;
- la précision ;
- la responsabilité intellectuelle ;
- la capacité à assumer ses hypothèses.

Le système ne cherche pas à valoriser :
- le clash ;
- le sensationnalisme ;
- le hot take ;
- la viralité émotionnelle.

---

# Définition d'une take

Une take est un objet structuré contenant :

- une affirmation ;
- un ou plusieurs auteurs ;
- une échéance ;
- un protocole de résolution ;
- des moyens de preuve ;
- éventuellement un engagement économique ;
- éventuellement un ou plusieurs challengers ;
- les règles de redistribution associées.

---

# Principes fondamentaux

## 1. Une take doit être résoluble

Une take doit prévoir :
- une échéance ;
- un protocole de résolution ;
- des preuves attendues ;
- une méthode d'interprétation.

Une take insuffisamment définie peut être :
- refusée ;
- suspendue ;
- invalidée.

---

## 2. Le protocole de résolution fait partie intégrante de la take

Une take n'est pas uniquement une opinion.

C'est également :
- un protocole ;
- un cadre méthodologique ;
- un mécanisme de vérification ;
- une définition préalable des preuves acceptées.

---

## 3. Une preuve doit être acceptable par les parties engagées

Le protocole de résolution doit satisfaire :
- le porteur ;
- le challenger éventuel ;
- les autres parties engagées.

L'équité ne doit pas être reconstruite après coup.

Elle doit être construite :
- avant le scellement ;
- avant le dépôt ;
- avant l'engagement économique.

Une take ne doit pas être considérée comme valide tant que :
- les preuves ;
- les critères ;
- les méthodes ;
- les seuils ;
- les sources ;

ne sont pas explicitement acceptés.

---

## 4. Le débat doit se déplacer vers les preuves

Le projet considère que :
- débattre des preuves est plus structurant que débattre des intentions ;
- une hypothèse mal mesurable est une hypothèse faible ;
- une hypothèse sérieuse doit accepter ses conditions de falsification.

Une partie importante de l'activité sociale du système peut donc porter sur :
- la qualité des preuves ;
- la construction des protocoles ;
- les moyens de vérification ;
- les méthodes de mesure.

---

## 5. La précision d'une take n'est évaluée qu'à travers son protocole

La plateforme ne cherche pas :
- à interpréter l'intention implicite ;
- à reconstruire le sens réel d'une affirmation.

Une take est évaluée uniquement :
- selon les critères déposés ;
- selon les preuves prévues ;
- selon le protocole accepté.

---

# Types de résolution supportés

Le système doit permettre plusieurs niveaux de preuve.

Les mécanismes de résolution peuvent inclure :

- données publiques ;
- API ;
- statistiques officielles ;
- résultats électoraux ;
- cours financiers ;
- sondages ;
- audits ;
- constats d'huissier ;
- preuves notariales ;
- arbitrages définis à l'avance ;
- protocoles tiers complexes.

Les moyens de résolution complexes doivent être supportés nativement.

---

# Validation des takes

La plateforme peut utiliser :
- des systèmes automatisés ;
- des robots de validation ;
- des systèmes hybrides ;
- des validations humaines limitées.

Une take peut être :
- refusée ;
- suspendue ;
- invalidée ;

si :
- elle est trop vague ;
- les preuves sont insuffisantes ;
- le protocole est contradictoire ;
- la résolution est irréaliste ;
- les critères sont inexploitables.

Les décisions doivent être motivées.

---

# Engagement économique

Les engagements économiques sont conçus comme :
- un coût de conviction ;
- une preuve d'engagement ;
- une friction volontaire ;
- une immobilisation assumée.

Le système n'est pas conçu comme :
- un investissement ;
- un produit financier ;
- un mécanisme de spéculation.

Le gain financier ne constitue pas le moteur principal du système.

---

# Gestion du coût d'opportunité

Le coût d'opportunité fait partie de l'engagement.

Le projet considère que :
- la perte potentielle de rendement ;
- l'immobilisation temporaire ;
- la perte de liquidité ;

participent à la preuve de conviction.

La plateforme ne promet pas de rendement financier.

---

# Montants engagés

Les montants engagés sont publics.

Le projet considère que :
- le coût visible ;
- le sacrifice assumé ;
- l'engagement affiché ;

participent à la crédibilité de la take.

---

# Challengers et takes opposées

Le système permet :
- de challenger une take ;
- d'exiger un engagement ;
- de pousser à la formalisation ;
- d'opposer plusieurs hypothèses incompatibles.

Exemple :
- "Le candidat X remportera l'élection."
- "Le candidat Y remportera l'élection."

Les parties peuvent :
- partager les mêmes preuves ;
- partager le même protocole ;
- partager la même échéance.

---

# Nature du challenge

Le challenger :
- n'est pas nécessairement un adversaire ;
- peut chercher une certification ;
- peut chercher une clarification ;
- peut chercher une formalisation.

Le challenge ne doit pas être gratuit.

Le système cherche à éviter :
- les demandes abusives ;
- les mécanismes de harcèlement ;
- les asymétries sans coût.

Le challenger peut :
- déposer une somme ;
- supporter un coût de contestation ;
- participer au protocole.

---

# Redistribution des pertes

En cas d'échec d'une take :
- les fonds peuvent être redistribués à des causes validées par la plateforme ;
- les redistributions peuvent être composites ;
- plusieurs causes peuvent recevoir des pourcentages différents.

Le système des causes bénéficiaires constitue un sous-système à part entière.

---

# Principes concernant les causes bénéficiaires

Le projet considère qu'affecter de l'argent à une cause :
- revient indirectement à ne pas financer d'autres causes ;
- modifie les arbitrages privés et publics ;
- influence la répartition réelle des ressources.

Le système doit donc chercher :
- une neutralité relative ;
- une gouvernance explicite ;
- des mécanismes transparents ;
- une limitation des détournements idéologiques.

Le projet refuse :
- les redistributions librement auto-dirigées ;
- les usages assimilables à du blanchiment ;
- les mécanismes d'optimisation fiscale implicite.

---

# Modèle économique

Le projet n'est pas conçu comme une plateforme spéculative.

Les modèles économiques envisagés incluent :
- frais fixes ;
- coûts de dépôt ;
- résolution complexe payante ;
- archivage ;
- certification ;
- services premium ;
- stockage long terme.

Le projet cherche à éviter :
- les modèles basés sur la spéculation ;
- la dépendance directe aux montants immobilisés ;
- les dynamiques assimilables à un casino.

---

# Pseudonymat et conformité

Le pseudonymat est autorisé.

Cependant :
- la plateforme doit rester conforme à la loi ;
- les obligations réglementaires applicables doivent pouvoir être respectées ;
- des mécanismes de contrôle doivent exister même s'ils ne sont pas publics.

---

# UX et paradigme produit

Le projet privilégie :
- la fiche structurée ;
- le registre ;
- l'archive ;
- la capsule temporelle ;
- l'objet documentaire.

Le paradigme de réseau social classique est évité.

La plateforme n'a pas vocation à devenir :
- un feed d'actualité ;
- un système de dopamine sociale ;
- une économie de réactions instantanées.

- une infrastructure ;
- un protocole ;
- un registre d'engagements ;
- une mémoire des hypothèses.

Le projet n'est pas :
- un média ;
- un forum ;
- un système de paris ;
- un réseau social classique.

Le projet cherche à limiter :
- les mécanismes assimilables au gambling ;

---

- explicite ;

# Effet recherché

> une affirmation publique gagne en valeur lorsqu'elle accepte :
> - une méthode de vérification ;
> - un horizon temporel ;

---

- lourdeur procédurale ;

- produit financier implicite ;
